Core truth Executor does not own MCPs. The Code Mode host does.

This map separates the combined Code Mode MCP server, the host-side MCP registry, the executor RPC callback host, and the sandbox. The sandbox receives provider stubs and a callback URL. Real MCP calls stay in the host runtime.

Local V1: one local Code Mode host, one local sandbox process per run

In v1, everything runs on the developer machine. The combined MCP server owns the public MCP surface. The registry owns upstream MCP clients. The executor only runs code and routes sandbox callbacks.

Who owns what?

If one sentence should stick: MCP authority is in the Code Mode host; sandbox authority is only the explicit provider capability it was handed for this execution.

Code Mode MCP server Owns public search/execute and turns registry metadata into model-facing APIs.
MCP Registry Owns upstream MCP client connections, tool discovery, original names, credentials, and dispatch.
Executor RPC host Owns callback routing for active sandbox runs. It does not discover MCP tools.
Sandbox runtime Owns generated JavaScript execution. It receives stubs, not MCP clients, env vars, or tokens.

Request trace: from execute to real MCP call

This is the actual mental model for an execute call. The sandbox never talks to upstream MCP servers directly; it talks to the executor callback URL, and the host handler dispatches through the registry.

User harness calls the combined MCP server

The outside client only sees the wrapper tools.

execute({
  code: "async () => github.create_issue({ title: 'Bug' })"
})

Code Mode asks the registry what tools exist

The registry has already connected to upstream MCP configs and preserved original names.

McpRegistry.listTools()

Code Mode creates executor providers

Each provider function is a host-owned handler backed by registry dispatch.

providers: [{
  name: "github",
  fns: {
    create_issue: (input) =>
      McpRegistry.callTool({
        server: "github",
        tool: "create_issue",
        arguments: input
      })
  }
}]

Executor starts sandbox and gives it a callback URL

The sandbox gets generated code, provider manifest, run URL, and bearer token.

http://127.0.0.1:PORT/runs/RUN_ID

Sandbox calls the provider stub

The generated JS thinks it is calling an API object, but it is really making an HTTP RPC callback.

await github.create_issue({
  owner: "acme",
  repo: "app",
  title: "Bug"
})

RPC host invokes the registered handler

The handler jumps back into the registry, which calls the real upstream MCP tool.

sandbox -> RPC host -> McpRegistry.callTool -> upstream MCP

Result returns to the sandbox and then to the user harness

The host keeps secrets and MCP clients. The sandbox only gets the tool result.

Remote future: same callback idea, different machines

Remote support does not change the ownership model. It only changes where the Code Mode host and sandbox run, plus how the sandbox reaches the callback URL.

Remote Code Mode host

Runs the combined MCP server, registry, credentials, policies, and provider handlers.

  • Can connect to remote-accessible MCP servers.
  • Can run configured stdio MCPs on the remote host machine.
  • Publishes or tunnels a callback URL for sandboxes.

Remote sandbox

Runs generated JS somewhere else and calls back to the host URL.

  • Receives provider manifest.
  • Receives run token.
  • Does not receive upstream MCP credentials.

Upstream MCP location

MCP servers live wherever the Code Mode host can reach them.

  • Remote HTTP MCP works if reachable.
  • Remote stdio MCP runs on the host machine.
  • User-local stdio needs a bridge.

Can host and sandbox both be remote?

Yes. The sandbox only needs a callback URL it can reach. The host needs access to the upstream MCP servers. Those are separate networking questions.

Works conceptually

Remote Code Mode host plus remote sandbox is clean when all upstream MCP servers are either hosted remotely or configured on the remote host.

Does not magically work

A remote Code Mode host cannot spawn or connect to stdio MCP processes on the user's laptop unless a local connector/bridge is running there.

Remote host + user-local MCPs: the missing bridge

This is the main future architecture fork. If the combined Code Mode server is deployed remotely but the user wants local stdio MCPs, something local must represent those MCPs to the remote host.

Deployment cases

Think about where the Code Mode host runs first. That determines which upstream MCP configs are meaningful.

Local host + local sandbox

V1. Can spawn local stdio MCPs and run local sandbox callbacks on localhost.

Remote host + remote sandbox

Future. Works for MCPs the remote host can reach or spawn.

Remote host + local sandbox

Possible only if local sandbox can reach the host callback URL and receives a token.

Remote host + local MCPs

Needs a connector. Without it, remote host has no path to user-local stdio processes.

Glossary: stop overloading "host"

Use these names in tickets and code comments so the architecture stays understandable.

Code Mode host

The process running the combined MCP server, registry, credentials, and policies.

Executor RPC host

The callback HTTP surface used by sandboxes to invoke host-owned provider handlers.

Sandbox runtime

The isolated code runner. It gets code, safe globals, provider stubs, URL, and token.

Upstream MCP server

The real side-effect layer. The registry connects to it and calls its tools.

Current implementation target: local Code Mode host plus local sandbox. Remote executor and local bridge support are future architecture, not shipped v1 behavior.
