Architecture Overview

Code Mode provides a type-safe bridge between generated sandbox code and upstream MCP tool providers. By utilizing a Build-Once-Stitch-Often caching strategy, it ensures that model discovery remains extremely fast while maintaining precise token window control.

Core: Effect-TS Service
Optimization: Declaration Caching
Engine: MCP Engine

Latest Updates

    Async schema compilation moved to at startup.

  • Sync Search

    search() is now a synchronous, pure stitching operation from cache.

  • Fast Path Cache

Design Rationale & Performance

Optimized Fragment Stitching

RESOLVED

The previous bottleneck of regenerating .d.ts on every search has been eliminated.

The Solution:

All heavy json-schema-to-typescript work happens once during buildDeclarationIndex. The DeclarationIndex stores pre-compiled "fragments" (interfaces and signatures) in memory.

Performance Result:

"Runtime search latency reduced from ~500ms+ (variable) to <1ms (constant), as it now only involves array filtering and string joining."

Precomputed Full Context

O(1) OPTIMIZED

The "Empty Query" fast path has been promoted to a primary data structure.

Mechanism:

The CodeModeRuntime now holds a fullContext property, pre-built during service initialization. Any search without a query simply returns this object reference.

Impact:

Eliminates all work for the most common discovery calls, reducing total server overhead during large-scale model interactions.

How are Interfaces Linked?

The "Glue" is the toolKey. The system ensures that a function never travels without its types by packaging them into a single Fragment Object at startup.

1

The Packaging

buildDeclarationIndex compiles the schema and saves the resulting interface code directly into the CachedToolDeclaration map.

2

The Filter

search() returns a list of matching tool keys. It doesn't know about code yet, just keys like "github.echo".

3

The Stitch

renderDeclarations iterates those keys. For each key, it pulls the entire package (Interfaces + Signature) from the map and joins them.

Updated API Reference

Function Lifecycle Description
buildDeclarationIndex STARTUP (Async) Compiles all tool schemas once into a fragment cache.
renderDeclarations RUNTIME (Sync) Pure function that stitches cached fragments into a final .d.ts string.
search RUNTIME (Sync) Filters tools and calls renderDeclarations. Zero async overhead.

The New Caching Lifecycle

1. The Build Phase (Startup)

  1. 1

    Discovery & Grouping

    McpRegistry returns tools; groupDiscoveredMcpTools organizes them by server.

  2. 2

    Index Building

    buildDeclarationIndex runs async compilation for every tool schema.

  3. 3

    Full Cache Warming

    fullContext is precomputed using renderDeclarations and stored on the Runtime.

2. The Search Phase (Runtime)

  1. 1

    Filtering

    filterCodeModeServers identifies the relevant tools for the query.

  2. 2

    Fragment Retrieval

    renderDeclarations pulls pre-compiled fragments from the DeclarationIndex.

  3. 3

    Sync Stitching

    Fragments are joined into a single string and returned. No I/O, no async work.
