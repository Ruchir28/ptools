MCP auth workflow in ptools

This doc is about upstream MCP auth — OAuth / bearer tokens for upstream MCP servers like GitHub, Notion, etc. It is not about a ptools user login system.

@ptools/auth defines contracts (CredentialsStore, AuthCoordinatorCore, …) and the shared state machine Layer. Host packages provide implementation Layers that satisfy those contracts.

Rule of thumb: auth_status and refresh belong to Code Mode runtime. The user-facing auth URLs belong to the host auth layer.

        3) How the auth flows work

        Model host → auth_status / refresh

        1. The model host calls @ptools/mcp-server.
        2. The MCP adapter forwards auth_status or refresh to the Code Mode client/runtime.
        3. Code Mode asks the registry for authStatus or refresh.
        4. The registry reads the shared AuthCoordinator.
        5. AuthCoordinatorCore returns the auth center URL, server statuses, and any authorize/setup URLs.

        Node browser auth

        1. NodeAuthCoordinatorLive starts a local auth center on localhost.
        2. PtoolsAuthManager serves /auth, /auth/:server, /setup, /refresh, and the callback route.
        3. PtoolsOAuthProvider stores OAuth material in the OS keyring.
        4. NodeMcpConnectorLive attaches OAuth only when the HTTP upstream actually needs it.

        Cloudflare browser auth

        1. The Worker verifies the public bearer token for auth/config routes.
        2. Worker routes call typed Durable Object RPCs.
        3. CloudflareOAuthFlow signs state and stores pending state in DO storage.
        4. The DO stores tokens, client info, discovery state, and PKCE verifier in DO storage.
        Cloudflare limitation: HTTP upstream MCP only for the first release. Stdio is intentionally rejected.

        6) Review checklist

        1. Start with shared contracts in packages/auth/src/index.ts and packages/auth/src/coordinatorCore.ts. These define the core boundary every host must implement.
        2. For any host change, name the shared contract it provides (CredentialsStore, AuthCoordinatorPolicy, etc.) before reading implementation details.
        3. Check status / URL / provider-cache behavior in AuthCoordinatorCoreLayer first — host layers should not reimplement that state machine.
        4. Then check the host adapter: packages/host-node/src/auth.ts or packages/host-cloudflare/src/layers/auth/*.
        5. For auth-route changes, review packages/host-node/src/auth.ts and packages/host-cloudflare/src/worker/routes/mcpAuth.ts.
        6. For upstream connection changes, review packages/host-node/src/mcpConnector.ts, packages/host-cloudflare/src/layers/mcpConnector.ts, and packages/mcp-registry/src/connect.ts.
        7. For model-facing tools, review packages/mcp-server/src/server.ts and packages/code-mode/src/CodeMode.ts.
        One-line mental model: @ptools/auth owns contracts + the shared state machine; hosts own platform Layers that provide those contracts; Code Mode only exposes status/refresh to the model host.
